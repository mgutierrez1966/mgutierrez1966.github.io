NOMBRE
tramitarSolicitudManto
VERSION
2
URL
/tramitarSolicitudManto
DESCRIPCIÓN
Realiza una determinada tramitación en una solicitud de mantenimiento. La tramitación modifica el estado de la solicitud, y puede realizar acciones adicionales, definidas en el flujo de proceso de la solicitud de mantenimiento.
DATOS ENTRADA
NOMBRE
TIPO
REQUERIDO
DESCRIPCION
solicitudMantenimiento
ID
SI
Identificador de la entidad SolicitudMantenimiento que se v a tramitar
tipoTramite
Entero
SI
Tipo de tramitación solicitada (ver TABLA 1, columna "Codigo")
observaciones
Texto
NO
Observaciones a la tramitación
accionesCorrectivas
Texto
NO
Acciones correctivas realizadas (para tramite tipo 2- Resolver por OACISE)
DATOS SALIDA
NOMBRE
TIPO
DESCRIPCION
PROCESO
  • Se creará un nuevo objeto SolicitudMantenimientoTramit, asociado a la solicitud de mantenimiento.
    - El atributo "estadoInicial" se establecerá al estado actual de la solicitud de mantenimiento
    - El atributo "estadoFinal" se inicializará según se indica en la TABLA 1 para cada trámite.
    - La fecha y hora se establece a la fecha y hora del sistema.
    - Las observaciones se inicializan con el valor del parámetro de entrada.
  • Se realizarán las acciones específicas indicadas para cada trámite en la TABLA 2

RESTRICCIONES
PRECONDICIONES
  • Se verificará que el tipo de solicitud, el rol de usuario y el estado actual de la solicitud se corresponde con lo indicado en la TABLA 1. Si no es así, se mostrará un error.
  • Si el tipo de solicitud es SolicitudMantAdaptaHw (adaptativo), y el trámite es "6 - Aprobar por Director ", la solicitud debe de tener asociado el almacén de baja (atributo "almacenBaja").
** INICIO ARIGIS-1083
  • Si el tipo de solicitud es SolicitudMantAdaptaHw (adaptativo), SolicitudTransporte (transporte) o SolRetiradaBajaHw (retirada para baja), y el trámite es "6 - Aprobar por Director ", todos los recursos asociados a la solicitud deben tener el peso asignado (atributo "peso")
** FIN ARIGIS-1083

 
POSTCONDICIONES
 - La solicitud de mantenimiento se encuentra en el estado final adecuado (ver TABLA 1)
- Se han creado las notificaciones a los usuarios implicados (ver TABLA 2)

TRANSACCIONAL
SI
AUDITADO
SI
URL
api/{NombreClaseAPI}/{NombreMetodo}
DESCRIPCIÓN
Crea una nueva solicitud de mantenimiento perfectivo a partir de unos datos básicos y de una lista de recursos asociados.
PARÁMETROS SOLICITUD (U:URI,B:BODY, H:Cabecera/Header)
NOMBRE
U/B/H
TIPO
REQUERIDO
DESCRIPCIÓN
solicitud
B
Objeto JSON
SI
Objeto de tipo SolicitudMantenimientoPerfectivo para dar de alta
recursos
B
Array de String
SI
Array de identificadores de recursoHW que deben asociarse a la solicitud
RESPUESTA
TIPO
JSON
NOMBRE
TIPO
DESCRIPCIÓN
nuevaSolicitud
Objeto JSON
Objeto SolicitudMantenimientoPerfectivo creado, con los campos inicializados, incluyendo el código asignado a la nueva solicitud.
PROCESO
  • Una vez validadas las precondiciones, la nueva solicitud se inicializa con los siguientes datos:
  • Estado :
    - Si el usuario es CISPOC: NUEVA
    - Si el usuario es DIRECTOR: Tramitada a Director por Subdirector.
  • Fecha de creación: fecha del sistema.
  • cispocCreador:  usuario que realiza la petición (aunque sea el Director).
** INICIO ARIGIS-902
  • nodoCispoc:
    - Si el usuario es CISPOC: NodoCargoRecursos apoyado por el CISPOC que realiza la petición.
    - Si el usuario es DIRECTOR: NodoCargoRecursos a cuyo cargo se encuentran los recursos.
** FIN ARIGIS-902
  • cecomOacise:
    - Para rol CISPOC: CECOM del OACISE que apoya a la UCO asociada al CISPOC
    - Para rol DIRECTOR: sin valor
  • Código: formato AAAANNNNNN, donde:
    - AAAA: es el año de la fecha del sistema.
                      - NNNNNN: es un número correlativo de 6 dígitos dentro del año.
          ** INICIO ARIGIS-559
         -       rolCreador: rol activo del usuario que realiza la solicitud.
          ** FIN ARIGIS-559

 
  • Se asocian los recursos a la nueva solicitud, llamando al método MantenimientoHwService.AgregarRecursosPerfectivo(), pasando la solicitud y los recursos a agregar.
  • Si el rol de usuario es CISPOC:
  • Se genera una notificación del tipo "Nueva solicitud de mantenimiento perfectivo", para los usuarios del OACISE que debe tramitar la solicitud (ver documento asociado a la entidad TipoNotificacion).
 
  • Si el usuario creador es DIRECTOR:
  • Tras la creación de la solicitud, se procede a su tramitación a la empresa, llamando a la API MantenimientoHwAPI.tramitarSolicitudManto, con el trámite '6 - Aprobar por Director'. Cualquier error devuelto por la API será comunicada al usuario.
 
DEPENDENCIAS
ANÓNIMO
AUTENTICADO
ROLES
SEGURIDAD DE ACCESO
X
CISPOC, DIRECTOR
PROCESO DIFERENCIADO POR ROL
SI (ver descripción proceso)
ROL
VARIACIÓN RESPECTO AL PROCESO
RESTRICCIONES
PRECONDICIONES
 - Para el rol CISPOC, la UCO apoyada por el CISPOC debe de tener un OACISE asignado.
- Para el rol CISPOC, todos los recursos a agregar deben ser accesibles para el usuario que realiza la solicitud.
- Todos los recursos deben estar a cargo de la misma UCO.
** INICIO ARIGIS-581
- El contrato de mantenimiento asociado a la nueva solicitud debe ser el mismo que se obtiene mediante la operación "MantenimientoHwService.ObtenerContratoMantenimientoAbierto()". Si esta operación devuelve NULO o devuelve un contrato de mantenimiento diferente, se devuelve el error "El contrato de mantenimiento especificado no es válido o no está en vigor".
** FIN ARIGIS-581
** INICIO ARIGIS-946
- No se permitirá crear la solicitud si alguno de los recursos incluidos ya tiene cualquier otra solicitud de mantenimiento en curso, salvo si es de tipo correctivo.
** FIN 946
